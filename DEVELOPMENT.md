# Yarncha Development Memory

This file is the project memory card for AI-assisted development. Before starting new work, read this file first. After every major change, update this file so future maintenance does not forget the current structure.

## App Purpose

Yarncha is a personal yarn craft project tracker and toolkit.

It helps users manage knitting, crochet, and Tunisian crochet projects. Core features include project tracking, chart/PDF upload, Reading Mode, annotations, Row Mask, Toolkit calculators, yarn stash, library, settings, and PWA support.

The product direction is local-first, mobile-first, and focused on daily craft workflow rather than adding many unfinished features.

## Tech Stack

- Framework: No component framework currently. The app remains a single-page app built with plain HTML, CSS, vanilla JavaScript, and small TypeScript cloud modules compiled by Vite.
- Styling system: Plain CSS in `styles.css`, using CSS variables/design tokens for themes and dark mode.
- Icon system: theme-aware minimalist outline SVGs. Static navigation icons live inline in `index.html`; dynamic icons are generated by `uiIcon()` in `app.js`. Avoid emoji or platform-dependent Unicode artwork in controls.
- State management: In-memory `state` object in `app.js`, loaded and saved manually.
- Storage method:
  - `localStorage` for the main app state JSON.
  - IndexedDB for larger file/blob assets such as uploaded charts, PDFs, library files, cart images, material photos, and project cover images.
  - Supabase Auth, Postgres and private Storage for opted-in private-beta cloud sync.
  - Local storage remains the first write and offline draft; cloud sync is debounced after authentication.
- Build tool: Vite. `npm run dev`, `npm run build`, and `npm run preview` are the supported cloud-beta commands.
- Production static copy: `scripts/copy-static.mjs` copies only the small classic app/PWA assets after Vite builds and validates `dist/index.html`; it has no dependency on a `vendor` directory.
- Legacy static preview: `python3 -m http.server 4183 --bind 0.0.0.0` still loads the core local app, but cloud TypeScript modules require Vite.
- Backend: Supabase migrations, Row Level Security, private Storage, and Edge Functions under `/supabase`.
- PWA files:
  - `public/manifest.json`
  - `service-worker.js`
  - `public/service-worker.js`
  - `manifest.webmanifest`
  - `sw.js`
  - icon files in `public/icons`
  - root icons: `favicon.ico`, `apple-touch-icon.png`, `apple-touch-icon-precomposed.png`, `icon.svg`
- Known libraries:
  - `pdfjs-dist` from npm; Vite emits the PDF worker as a production asset.
  - `tesseract.js` from npm; OCR worker/core/language data load on demand from jsDelivr and Project Naptha rather than Git.
  - `@supabase/supabase-js` from npm.

## Calculator Engine Rules

- Keep calculator formulas in `calculator-engine.js` when possible.
- Calculator outputs should include:
  - formula
  - variables
  - intermediate calculations
  - final outputs
  - confidence score
  - manual verification warning when confidence is below 90
- Do not round intermediate calculations for storage; round only displayed final values.
- Do not claim professional certainty when the formula is an estimate. Use clear confidence and notes.
- Add or update tests in `tests/calculator-engine.test.js` when calculator formulas change.

- The cloud Account & Sync card must not include destructive actions. It should show email, sync status, sign-in/account access, sign out, and sync preferences only.
- Delete Account and cloud-data deletion live in a dedicated Danger Zone card with a red border, warning icon, descriptive text, and the exact warning copy `This action cannot be undone.`
- Account deletion remains disabled until the user types `DELETE` or confirms the signed-in email address.
- Keep future destructive Settings actions inside Danger Zone rather than mixing them with normal cloud sync controls.
- `npm run test:settings` covers the Settings grouping and deletion-confirmation contract.
- The Vite production build intentionally strips Yarncha's three classic root scripts before Vite's HTML transform, then `scripts/copy-static.mjs` injects them back into `dist/index.html` after copying static assets. This keeps the vanilla runtime intact while avoiding misleading non-module script warnings.
- Classic app assets currently use cache key `v=49`; bump the root `index.html` scripts and `scripts/copy-static.mjs` together whenever browser previews show stale root scripts.
- `openModal()` must call `ensureModalElements()` before writing modal content. Production module scripts can leave the static modal shell unavailable, so modal actions should self-heal instead of assuming `#modal-backdrop` and `#modal-content` already exist.
- Library cards should not expose visible destructive Delete actions. Yarn materials, generic Library pages/PDFs, and Project Ideas keep normal card actions only; deletion lives inside each Edit modal's small Danger Zone with the confirmation text `Are you sure you want to delete this item?`. `npm run test:library` covers this contract.
