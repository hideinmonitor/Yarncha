# Yarncha Development Memory

This file is the project memory card for AI-assisted development. Before starting new work, read this file first. After every major change, update this file so future maintenance does not forget the current structure.

## App Purpose

Yarncha is a personal yarn craft project tracker and toolkit.

It helps users manage knitting, crochet, and Tunisian crochet projects. Core features include project tracking, chart/PDF upload, Reading Mode, annotations, Row Mask, Toolkit calculators, yarn stash, library, settings, and PWA support.

The product direction is local-first, mobile-first, and focused on daily craft workflow rather than adding many unfinished features.

## Tech Stack

- Framework: No component framework currently. The app remains a single-page app built with plain HTML, CSS, vanilla JavaScript, and small TypeScript cloud modules compiled by Vite.
- Styling system: Plain CSS in `styles.css`, using CSS variables/design tokens for themes and dark mode.
- State management: In-memory `state` object in `app.js`, loaded and saved manually.
- Storage method:
  - `localStorage` for the main app state JSON.
  - IndexedDB for larger file/blob assets such as uploaded charts, PDFs, library files, cart images, material photos, and project cover images.
  - Supabase Auth, Postgres and private Storage for opted-in private-beta cloud sync.
  - Local storage remains the first write and offline draft; cloud sync is debounced after authentication.
- Build tool: Vite. `npm run dev`, `npm run build`, and `npm run preview` are the supported cloud-beta commands.
- Legacy static preview: `python3 -m http.server 4183 --bind 0.0.0.0` still loads the core local app, but cloud TypeScript modules require Vite.
- Backend: Supabase migrations, Row Level Security, private Storage, and Edge Functions under `/supabase`.
- PWA files:
  - `public/manifest.json`
  - `service-worker.js`
  - `public/service-worker.js`
  - `manifest.webmanifest`
  - `sw.js`
  - icon files in `public/icons`
  - root icons: `favicon.ico`, `apple-touch-icon.png`, `apple-touch-icon-precomposed.png`, `icon.svg`
- Known libraries:
  - `vendor/pdf.mjs`
  - `vendor/pdf.worker.mjs`
  - `vendor/tesseract/tesseract.min.js`
  - `vendor/tesseract/worker.min.js`

## Current Storage Logic

Main state is stored in `localStorage` under the key defined in `app.js`:

- `const STORAGE_KEY = "threadline-data-v1";`

Large files are stored in IndexedDB:

- Database name: `threadline-files`
- Version: `2`
- Object stores:
  - `files`
  - `state`

Important storage functions in `app.js`:

- `openAssetDb()`

- `getAsset(id)`
- `saveState()`
- `loadState()`
- `saveProjectTouch(project)`

Stored locally:

- projects
- project names and types
- preferred unit system (`unitSystem`: metric or imperial)
- current row
- planned rows
- sub counters
- repeat counters
- markers
- notes
- chart references
- uploaded chart/PDF/image assets
- annotations
- Row Mask data
- project cover image references
- yarn stash/inventory
- shopping cart
- budget settings
- library sections/items
- theme preference
- appearance style preference
- project idea board entries
- project idea reference image asset ids
- onboarding state

Exported backup files are JSON files generated by the app. They are the current manual way to move data between browsers/devices.

Current limitation:

Local-only saving is per browser and per device. Signed-in beta users can explicitly migrate local projects to Supabase and then sync across devices. Safari and Chrome still retain separate local draft copies. Clearing browser data can remove the local copy unless the user has exported a backup or completed cloud migration.

## Calculator Engine Rules

- Keep calculator formulas in `calculator-engine.js` when possible.
- Calculator outputs should include:
  - formula
  - variables
  - intermediate calculations
  - final outputs
  - confidence score
  - manual verification warning when confidence is below 90
- Do not round intermediate calculations for storage; round only displayed final values.
- Do not claim professional certainty when the formula is an estimate. Use clear confidence and notes.
- Add or update tests in `tests/calculator-engine.test.js` when calculator formulas change.
- Current integrated engine-backed tools:
  - Gauge / Swatch Calculator
  - Needle / Hook Adjustment
  - Pattern / Garment Resizer
  - Blocking Calculator
  - Hat / Beanie Size Calculator
  - Sock Calculator
  - Circle Calculator
  - Amigurumi Shape Guide
  - Granny Square Size Planner
  - C2C Blanket Calculator
  - Sleeve Calculator
  - Raglan Calculator
  - Blanket Calculator
  - Grid Generator
  - Stripe Generator
  - Color Pooling Planner
  - Yarn Estimator
  - Yarn Leftover Estimator
  - Yarn Substitution Helper
  - Yarn Weight Converter
  - Unit Converter

## Project Ideas / Idea Board Rules

- Project Ideas are inspiration records, not active projects.
- Do not automatically create a project when saving a calculator result as an idea.
- Ideas can later become projects through `Create Project from Idea`; the original idea should remain in the idea board.
- Ideas can store:
  - title
  - craft type
  - project kind
  - inspiration notes
  - source platform
  - source link
  - reference image asset id
  - saved calculator results
  - yarn estimate
  - difficulty guess
  - tags
  - archived state
  - created/updated dates
- Calculator result cards should continue supporting:
  - Copy Result
  - Save to Notes
  - Link to Project
  - Link to Idea
  - Save as Project Idea
  - Add to Buy List when relevant
- Read `TOOL_SYSTEM_IMPLEMENTATION.md` before major Toolkit work.
- Project Toolkit UX currently uses a compact Category selector plus Tool selector on project pages. Do not reintroduce a giant flat project-tool dropdown. The master Tools page can still show the full toolbox.
- Rendering tools are merged in project UX as Project Rendering Studio with Grid, Stripes, and Color Pooling tabs.
- Budget Calculator must remain out of Project Toolkit; budget belongs in Buy List / Budget and calculator results can flow into Buy List.

## Development Rules For Future AI Edits

- Read `DEVELOPMENT.md` before starting new work.
- Do not place all new logic into one huge file forever. The current app is already concentrated in `app.js`; future major features should be split carefully.
- Split large features into components/modules when it is low risk and clearly improves maintainability.
- Keep mobile-first design.
- Test at 390px, 430px, 768px, and desktop.
- Do not add OAuth, Supabase, Google login, Apple login, backend sync, or custom domain unless explicitly requested.
- Do not make AI chart recognition the main workflow unless explicitly requested.
- Do not rewrite unrelated features when fixing one bug.
- Do not remove existing local data fields without a migration plan.
- Preserve local-first behavior unless the user explicitly asks for backend/cloud work.
- Update `DEVELOPMENT.md` after every major change.
- Keep user-facing wording non-technical and friendly.
- Use clear empty states, error states, loading states, and touch targets of at least 44px.

## Git Safety Recommendation

Before major AI changes:

1. Commit the current working version.
2. Create a checkpoint.
3. Test the current app.
4. Then implement the next small task.

After each change:

1. Test in browser.
2. Test mobile layout.
3. Check console errors.
4. Confirm local data still survives refresh.
5. Update `DEVELOPMENT.md` if structure, storage, features, or constraints changed.

## Current Known Maintenance Notes

- This app now contains a Supabase/Vercel private-beta implementation, but it is not connected or deployed until project credentials, migrations, Edge Function secrets and Vercel authentication are supplied.
- The preview server is development-only; Vite is required for the cloud modules.
- Local storage remains browser/device-specific and is intentionally retained as an offline draft.

- PDF viewing uses browser/PDF.js behavior, not a full custom PDF editor.

- The current architecture works for MVP speed, but long-term maintainability will require gradual feature/module extraction.

## Current Toolkit UX Notes

The Project Toolkit uses a visual category-card workflow, not a long dropdown. The four project-facing categories are:

- Stitch & Pattern Tools
- Fit & Size Tools
- Project Rendering Studio
- Yarn Tools

Project pages filter tools by project craft type. Knitting projects show knitting + shared tools; crochet projects show crochet + shared tools. Budget-specific calculator work should stay in Buy List / Budget rather than appearing as a project toolkit category.

Calculator results are intentionally user-controlled. Results can be copied, saved to notes, linked to project Tool History, saved as Project Ideas, or added to Buy List when relevant. Do not auto-create projects or purchases from calculator results without a user action.
